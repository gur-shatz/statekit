# Metrics


## Intro
Statekit provides metrics and state aggregation.
States are aggregated and also a dashboard is available to display the state. Metrics are aggregated, and they are currently only available to scrape onwards as prometheus metrics.

I want to provide infrastructure that allows us in the dashboard to drill down from a target to the relevant metrics, and present a charts built out of metrics.

This means three things: Like state being aggregated, we can optionally create a metrics aggregator (e.g. where we would also have state storage), the metrics aggregator will have timeseries data built on the metrics, and the state dashboard will have a drill down into a target level dashboard of charts representign the metrics.

So we have
- metrics aggregator making timeseries from metrics, with API for generating timeseries for a **keyed** item.
- UI in the dashboard to drill down for a specific **keyed** item.
- UI for rendering charts of the items, grouped by metric (with different labels as groups)

## Metrics Aggregator

The metrics aggregator collects metrics **keyed** by the `scrape_path`.

**We should optimize memory consumption as much as possible**

- each metric under the label is saved in a timelimited timeseries (globally defined, default 30m).
- strings are kept separate from their identifier using a dictionary (see ../scandex for similar), and labels (postings) are encoded separately from measurements
- Timestamps are second based and. Timestamps are saved separtely from measurements.
- We cap the number of entries per metrics (avoid too many labels) by dropping the smallest values

## Questions:

- What is the estimated storage size for this?

With the implemented encoding, the timestamp/value payload is approximately:

`keys × metrics/key × series/metric × (retention / scrape interval) × 16 bytes`

The 16 bytes are one Unix-second `int64` and one `float64`. At the default
30-minute retention and a 15-second scrape interval, each series has 120
points (1.9 KiB). A key with 100 metrics and an average of 10 label sets per
metric therefore uses about 1.9 MiB of point payload, plus Go slice/map
overhead, encoded label postings, and the shared string dictionary. At the
default hard cap of 100 label sets per metric, that example is 19.2 MiB of
point payload per key. Slice spare capacity can temporarily approach 2x, and
a one-second scrape interval uses 15x the point storage.
